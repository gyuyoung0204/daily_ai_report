# 설계: `/quality-check` 품질 평가 스킬

**작성일:** 2026-06-12
**상태:** 승인됨 (구현 대기)

---

## 1. 목적

프로젝트를 품질 루브릭 기준으로 **에이전트가 채점 → 점수 진단 리포트 산출 → 목표 점수 미달 시 기존 스킬에 개선 위임 → 재채점**을, 목표 점수 도달까지 반복한다.

이 스킬은 **채점 두뇌**다. 직접 코드를 고치지 않고, 개선 작업은 기존 스킬(`/issue-processor`, `/documentation` 등)에 위임한다.

---

## 2. 핵심 결정 (브레인스토밍 합의)

| 항목 | 결정 |
|------|------|
| 산출물 | 점수 진단 리포트 (차원별 0~4점 + 근거 + 총점/등급 + 개선 제안) |
| 루프 | 목표 점수 도달까지 반복 |
| 개선 수행 주체 | 기존 스킬에 위임 (이 스킬은 채점만) |
| 채점 주체 | Claude 에이전트가 평가 |
| 평가 모델 | **이중 평가 — 정형(객관 지표) + 정성(에이전트 판단)** |
| 루브릭 저장 | `QUALITY_RUBRIC.md` 단일 파일 (단일 출처) |

## 3. 구성 요소 (기존 하네스 패턴 준수)

| 요소 | 경로 | 역할 |
|------|------|------|

| 스킬 (진입점) | `.claude/skills/quality-check/SKILL.md` | 루프 오케스트레이션. 인자: 목표 점수(기본 90), 최대 반복(기본 5) |

## 4. 이중 평가 모델 (정형 + 정성)

에이전트가 평가하되 **두 종류의 점수를 따로 산출**한 뒤 종합한다.

### A. 정형평가 (Quantitative) — 객관 지표, 재현 가능
grep · 파일 존재 · 이슈 상태 등 **기계적으로 확인 가능한 증거**로 채점. 같은 입력이면 누가 돌려도 같은 점수.

| 차원 | 정형 지표 (실제 측정) |
|------|----------------------|
| 기능 완성도 | `$rawNews` 하드코딩 배열 존재 여부, 실 API/RSS 호출 코드 유무, 인사이트의 수집 뉴스 연동 여부 |
| 견고성 | `try/catch` 블록 수, 하드코딩 절대경로 개수(`C:\tmpfile` grep), 종료코드 처리 유무 |
| 테스트 | `test_*.ps1` 파일 수, 테스트 통과/실패 |
| 자동화 | 스케줄 등록 여부, `logs/metrics.json` · `aggregated_summary.json` 존재 |
| 보안 | 토큰 `.gitignore` 등록 여부, 하드코딩 자격증명 grep 적발 수 |

### B. 정성평가 (Qualitative) — 에이전트 판단, 근거 서술
지표로 안 잡히는 차원을 에이전트가 읽고 0~4점 + 근거 인용.

| 차원 | 정성 판단 기준 |
|------|---------------|
| 데이터 신뢰성 | 출처 신뢰도, 기사 선정의 적절성 · 시의성 |
| 문서 품질 | 중복 없음 · 코드와 일치 · 진행률 정확성 |
| 결과물(다이제스트) | 가독성, 관련성, 사용자 가치 |

### C. 종합
리포트에 **정형 점수표 + 정성 점수표를 나란히** 출력하고, 가중 결합한 **종합 총점 · 등급**을 산출.
두 점수가 크게 벌어지면(예: 정형 高 / 정성 低) 그 **불일치를 리포트에 명시** → 개선 위임의 우선순위 근거로 사용.

---

## 5. 평가 차원 (7종) · 가중치

| # | 차원 | 가중치 | 평가 축 |
|---|------|--------|---------|
| 1 | 기능 완성도 | 25% | 정형 주도 + 정성 보조 |
| 2 | 데이터 신뢰성 · 정확성 | 20% | 정성 주도 + 정형 보조(중복 제거 동작 등) |
| 3 | 견고성 · 에러 처리 | 15% | 정형 |
| 4 | 테스트 · 검증 | 15% | 정형 |
| 5 | 자동화 · 운영 | 10% | 정형 |
| 6 | 문서 품질 | 10% | 정성 |
| 7 | 보안 | 5% | 정형 |

각 차원 Lv0~4 기준의 상세 정의는 `QUALITY_RUBRIC.md`에 둔다 (본 spec은 구조만 규정).

**등급:** 90–100 🥇 Production / 75–89 🥈 Solid / 60–74 🥉 Working / <60 🚧 Prototype

**총점 공식:** `Σ(차원점수 × 가중치) / 4 × 100`

## 6. 데이터 흐름 (루프)

```
/quality-check [목표=90] [최대반복=5]
   │
   ├─▶ quality-evaluator 에이전트 호출
   │      ├─ QUALITY_RUBRIC.md 읽기
   │      ├─ 정형 증거 수집 (grep / 파일존재 / 이슈상태 / 테스트실행)
   │      ├─ 정성 판단 (근거 인용)
   │      └─ 정형 점수표 + 정성 점수표 + 종합 총점/등급
   │
   ├─▶ logs/quality_scores.json 에 회차 기록
   │
   ├─ 종합 총점 ≥ 목표?  ── 예 ──▶ 최종 리포트 출력 후 종료 ✅
   │       │
   │       └─ 아니오
   │
   ├─▶ 최저 가중점수 차원 1~2개 선정 → 개선을 기존 스킬에 위임
   │      · 기능/데이터/테스트/견고성/보안 약점 → /issue-processor
   │      · 문서 약점 → /documentation
   │
   └─▶ 재채점 (최대 반복까지). 점수 정체(2회 연속 미상승) 시 사용자 보고 후 중단
```

---

## 7. 설계 원칙 매핑 (프로젝트 3원칙)

- **중복 없이:** 루브릭은 `QUALITY_RUBRIC.md` 한 곳에만. 스킬 · 에이전트는 참조만.
- **사실 기반:** 에이전트는 **실제 파일/이슈/테스트 결과를 근거로 인용**해야 점수 부여. 근거 없는 점수 금지. 정체 시 임의 진행 대신 사용자 보고. 정형 지표가 곧 인용 가능한 증거.
- **정확함:** 각 차원 점수에 구체적 파일명 · 라인 · 이슈번호 명시.

---

## 8. 안전장치

- **무한루프 방지:** 최대 반복 횟수(기본 5) + 점수 정체(2회 연속 미상승) 시 자동 중단 · 보고.
- **역할 분리:** 이 스킬은 직접 코드를 고치지 않음. 개선은 기존 스킬에 한정 위임.
- **인코딩:** 한글 포함 `.ps1`/`.md`는 UTF-8 BOM (기존 add-bom 훅 처리).

## 10. 테스트

- **스모크 테스트:** `quality-evaluator`가 현재 프로젝트를 1회 채점해 정형/정성 점수 + 근거 포함 리포트를 산출하는지 확인.
- **스키마 검증:** `logs/quality_scores.json`이 위 스키마를 만족하는지 확인.
- **루프 안전장치 검증:** 점수 정체 시 정상 중단 · 보고하는지 확인.

---

## 11. 범위 밖 (YAGNI)

- 품질 게이트(통과/실패 차단) — 산출물은 진단 리포트로 한정.
- 정형 지표 수집을 위한 별도 PowerShell 스크립트 — 에이전트가 직접 grep/명령 실행으로 수집 (도구 추가 최소화).
- 점수 추이 시각화 대시보드 — 향후 별도 작업.
